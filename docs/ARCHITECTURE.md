# ARCHITECTURE

## 设计原则（为下一代兵棋而建）
1. **模块化**：单一确定性 Rust 内核 + 可插拔边界——规则即数据（`config/`）、概率 provider 可换（静态表/贝叶斯/学习模型）、绑定可选（PyO3 / wasm / native server）、资源（地图/想定/规则）皆可编辑可替换。加新规则、新算子、新资源都不动核心架构。
2. **智能化**：原生面向 AI/agent。对外提供结构化 JSON 工具接口（LLM 指挥官）与 Gym/PettingZoo 环境（RL）；同一内核支持人对人、人机、机机、以及**自博弈训练**。决策 tick 把实时连续时间离散成"可下令时刻"，便于 agent 接入。
3. **可信**：确定性（seed+指令流可重放）、战争迷雾正确（observe 只给可见信息）、契约先行（schema 唯一真源）、规则有权威真值（仓内 `config/tables/*.json`，经双人交叉核验）。错误不可累积互掩。

单一内核 = 单一真理。任何一端（训练、前端、服务器）看到的规则与裁决完全一致，杜绝"前端一套、训练一套"的漂移。

## 为什么是离散事件 + 决策 tick
规则是实时制：状态转换有固定时长（机动→停止 75s、压制 150s、间瞄 fly150s/boom300s/cd300s…），指令"先到先裁"。内核维护一个**按时间排序的事件队列**：
- `Engine::submit(cmd, t)` 在逻辑时刻 `t` 投递指令（同刻多指令按到达序裁决）。
- `Engine::advance_to_next_event()` 直接跳到下一个事件（无头高速）。
- `Engine::step(dt)` 推进 `dt` 仿真秒并处理到期事件（前端/对战用）。
- **决策 tick**：对 LLM/RL，按 `rules.timing.decision_tick_seconds` 把连续时间离散成"可下令时刻"。tick 之间引擎自由推进事件。

## 连续时间引擎约定（continuous-time）
**先澄清术语**：本内核是 **DES（离散事件模拟）**，而 DES 正是实现**连续时间**的标准手法，**不是**离散/回合时间。
- *离散/回合时间*：时间切成等长 Δt，事件只能落在 tick 边界。
- *本内核（连续时间）*：时间是实数量、事件带**精确时间戳**，引擎从一个事件**直接跳到下一个**；两事件之间状态不变，故任意时刻状态可精确求出。这与规则"双方连续观察、连续下令、**先到先裁**"、状态转换按**时长**（75/150/300s）、机动按"秒/格"完全对应——不是回合制。

本平台是 **连续时间 + 离散（六角）空间 + 事件驱动**：时间/下令时机/定时效果连续；空间是六角格、裁决在**格中心**结算（规则 1.b：机动不足 1 格以上一格为裁决基准），故内核**不需要亚格连续位置**，只需"t 出发 A 格 → t+dur 到 B 格"的到达事件 + 到达时重算通视/观察。该离散是规则自身的建模选择，非引擎缺陷。

### 1. 整数定点时钟（禁用 f64 计时）
- **时间键用整数**（建议厘秒 cs 或毫秒 ms 的 `i64` 逻辑时钟），**不得用 `f64`**。规则里 25/75/150/300、144/20/8/4 秒全是整数，1 格 = 200 m，按"秒/格"换算后仍是整数；整型时钟保证跨平台、可重放**完全确定**（这是硬规则#1 的直接落地）。
- 所有时长/间隔来自 `config`（rules-as-data），单位统一换算到时钟刻度后再入队，**逻辑层不出现浮点时间**。
- 距离=格数（整数）；速度查 `unit_speed`（整数秒/格）；到达时刻 = 出发时刻 + Σ(每格用时)，每格用时由地形/坡度/状态修正后**仍取整**（修正系数若产生分数，按规则定义的取整规则归一，不在时钟里留浮点）。
- `i64` 厘秒可表示约 ±29 亿年量级，溢出非现实问题；但所有时间运算用 `checked_add` 风格、不 panic（硬规则#4）。

```rust
/// 逻辑时钟刻度（厘秒）。内核所有时间都是它，绝不用 f64。
pub type Tick = i64;            // 1 秒 = 100 cs
pub const CS_PER_SEC: Tick = 100;
```

### 2. 事件抢占模型（取消 / 重排 / 失效）
实时规则里大量"中途打断"，连续时间引擎的真正硬核就是**事件可被取消或抢占**：
- *机动中被压制*：走完**当前格**才停（已调度的"到达当前目标格"事件**保留**，其后续各格的到达事件**作废**），到达后转被压制态。
- *收到停止令*：同样完成当前格、作废后续；并追加 75s 惩罚性"机动→停止"转换事件。
- *进入同格交战 / 炮兵机动取消间瞄点*：作废相关挂起事件，按新状态重排。
- 实现：每个事件带 `(time, seq)` 全序键 + 一个**有效性判定**。推荐二选一——
  - **代次号（generation）**：单位状态持一个 `gen: u64`，事件携带发起时的 `gen`；出队时若 `event.gen != unit.gen` 则丢弃（lazy invalidation）。改变机动计划即 `gen += 1`，旧事件自动失效，无需在堆里检索删除。
  - 或显式 `cancel(handle)` + 堆内 tombstone。
- **严格全序与稳定 tie-break**：出队比较键为 `(time, seq)`，`seq` 是**全局单调到达序号**；"先到先裁"= 同 `time` 时按 `seq` 升序。**禁止**以 `HashMap` 迭代序或单位 id 偶然顺序决定裁决先后（否则破坏确定性）。

```rust
#[derive(PartialEq, Eq)]
struct Scheduled { time: Tick, seq: u64, ev: Event }   // BinaryHeap<Reverse<Scheduled>>
// Ord: 先比 time，再比 seq；seq 由 Engine 单调发号，保证“先到先裁”可重放。
```

### 3. 决策 tick ≠ 仿真步长
两者**正交**，别混用：
- **仿真**始终连续推进（`advance_to_next_event` 瞬跳到下一个事件），与任何"步长"无关。
- **决策 tick**（`rules.timing.decision_tick_seconds`）只是**控制/轮询频率**——在这些时刻轮询 LLM/RL 下令；对应 RL 里 **SMDP / options** 的决策时刻（一次决策可跨越多个仿真事件）。tick 之间不打扰 agent，引擎自由跑。
- **人类**在 Web 端可在**任意时间戳**下令（真连续）；引擎一视同仁——人或 agent 的指令都只是带 `submit(cmd, t)` 时间戳的事件，无特殊路径。
- 因此：调 `decision_tick_seconds` 改的是"多久问一次 agent"，**不改仿真精度**；前端帧率/加速倍率改的是"放给人看的快慢"，也**不改逻辑结果**。

### 4. 逻辑时间 vs 墙钟时间（分离）
- 内核只认**逻辑时钟**（Tick），**与墙钟无关**。
- *无头训练/自博弈*：`advance_to_next_event()` 一路瞬跳，毫秒级跑完整局——没有"实时等待"。
- *真人对局*：前端把逻辑秒映射到墙钟，用 `step(dt)` 按帧推进或加速/暂停；这套节奏完全在前端一层，内核不变。
- 重放只复用 `seed + 有序指令流（含各自时间戳）`，与当初是瞬跑还是实时播放无关——逐 tick 状态必然一致（见「复盘」与 🔴 T0.2）。

### 落地不变量（实现这几条时必须守住）
1. 时间键 = 整数 `Tick`；内核计时**零 `f64`**。2. 出队全序 `(time, seq)`，`seq` 单调发号，"先到先裁"稳定可重放。3. 事件可失效（代次号/取消），打断=作废后续到达事件、保留"当前格"到达。4. `decision_tick` 是控制频率、`step(dt)` 是墙钟节奏，二者都不改仿真结果。5. 一切时长来自 `config`、不 panic。
> 这些主要由 🔴 任务坐实：**T0.2**（重放一致性内核）、**T1.1**（机动状态机/转换计时）、**T2.8**（同格交战的周期事件与抢占）。`engine.rs` 的事件堆时间键应显式定为 `Tick`（整型）。

## 确定性
- 唯一随机源：`rng::Rng`（PCG，显式 seed）。所有查表裁决从它取随机数。
- 状态里所有集合用有序结构（`BTreeMap`/`Vec`），不依赖 hash 顺序。
- 复盘 = `seed + 有序指令流`。重放即用同 seed 重跑指令流；可选周期性快照仅用于查看器快速 seek。

## 概率 provider（P3：静态表 / 贝叶斯校准 / 学习模型可插拔）
裁决不直接写死查表，而是经 `prob::ProbProvider`：
```rust
trait ProbProvider {
    fn resolve(&self, ctx: &ResolveContext, rng: &mut dyn Rng) -> Outcome;
}
```
`rules.prob.providers` 为每张表指定后端：`static` | `bayesian` | `model`。
- `static`：照 `config/rules.*.json` 里的裁决表查（与规则原文一致）。
- `bayesian`：每个"格子/结果"建模为 Dirichlet-Multinomial，先验来自专家指定或静态表，后验用观测数据更新（离线，见 `python/prob_learning`）。运行时仍是一次确定性采样。
- `model`：参数化模型（GLM/小网络）挂点，先留 stub。
"指定信息"两路：① 复盘/历史 outcome 数据；② 专家直接给概率/先验。

## 地图 vs 想定 vs 规则（三份独立契约）
- **地图**（`schemas/map.schema.json`）：静态地形——六角格、高程、地物（居民地/丛林/河/湖/松软/公路/铁路）、路网等级。
- **想定**（`schemas/scenario.schema.json`）：可变编成——双方单位与摆放、目标(夺控点)、胜负条件、时限、预置设施(雷场/路障/工事/间瞄炮火区)、单位初始状态。一图可配多想定。
- **规则**（`schemas/rules.schema.json` + `config/rules.default.json`）：所有数值与裁决表 + 概率 provider 配置。

## 复盘
`schemas/replay.schema.json`：`header`(双方名/地图名/想定名/seed/规则版本/引擎版本/时间/结果) + `commands`(canonical) + 可选 `snapshots`(查看器 seek 用) + `events`(日志)。查看器复用 `web/` 渲染层，时间轴拖动。

## 编辑器与资源模型
地图 / 想定 / 规则三类资源分**内置默认**（只读基线，含内置规则与跑通所需最小集）与**用户资源**（编辑器手动新建或从文本/JSON 导入，可导出分发）。三个图形化编辑器在 `web/src/editor/`，共用 `resourceIO.ts`（加载/导入/ajv 校验/导出）。契约 = `schemas/*.json`，与 Rust serde、`tools/validate.py` 共用。详见 `docs/RESOURCES.md`。
